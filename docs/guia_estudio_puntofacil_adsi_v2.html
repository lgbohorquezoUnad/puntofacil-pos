Guia de estudio de la aplicacion PuntoFacil POS

Version 2 - material de apoyo para ADSI del SENA

1. Presentacion

Esta version 2 fue escrita con un enfoque mas academico y explicativo. Su finalidad es ayudarte a estudiar programacion de software a partir de un proyecto real. En lugar de limitarse a nombrar archivos, esta guia explica la razon de ser de cada modulo, la relacion entre las capas del sistema y los conceptos tecnicos que puedes fortalecer como estudiante de ADSI.

La aplicacion analizada es PuntoFacil POS, un sistema web orientado a punto de venta, administracion, caja e inventario. Al estudiar este proyecto puedes ver claramente como se construye una solucion empresarial completa, desde la interfaz del usuario hasta la persistencia en base de datos.

2. Objetivo academico de la guia

  • Explicar como funciona la aplicacion de forma clara y precisa.
  • Describir los programas, lenguajes y librerias utilizados.
  • Identificar el contenido de cada carpeta y el proposito de sus archivos.
  • Relacionar el proyecto con conceptos clave de ADSI.
  • Servir como ejemplo de estudio para complementar tus clases y practicas.

3. Descripcion general del sistema

PuntoFacil POS es un sistema web que permite manejar procesos operativos y administrativos de un negocio. No se trata solamente de vender productos; tambien incorpora seguridad, control de inventario, trazabilidad y apoyo a decisiones operativas.

Entre las funciones principales del sistema se encuentran:

  • Inicio de sesion con usuarios del sistema.
  • Control por roles: administrador y cajero.
  • Registro de ventas desde una interfaz POS.
  • Apertura y cierre de caja.
  • Administracion de productos.
  • Administracion de usuarios.
  • Control de inventario con movimientos historicos.
  • Sugerencias de reposicion de stock.
  • Control de lotes proximos a vencer.
  • Ordenes de compra y recepcion de mercancia.
  • Conteos fisicos y conciliacion de inventario.
  • Transferencias entre ubicaciones.
  • Registros de auditoria.

4. Arquitectura del proyecto

La aplicacion sigue una arquitectura por capas. Esto es muy importante en ADSI porque enseña a separar responsabilidades. Cada capa cumple una tarea concreta y evita mezclar toda la logica en un solo archivo.

Flujo general: Usuario -> Interfaz web -> API Flask -> Routes -> Controllers -> Models -> MySQL

  • Frontend: es la parte visual que usa el usuario.
  • Routes: mapean URL a funciones.
  • Controllers: validan datos, revisan permisos y coordinan la logica.
  • Models: ejecutan sentencias SQL y manipulan datos.
  • Base de datos: almacena de forma persistente la informacion.

En terminos de analisis y diseño, esta organizacion mejora el mantenimiento, la escalabilidad y la claridad del codigo.

5. Programas, lenguajes y librerias utilizadas

5.1 Lenguajes

  • Python: lenguaje principal del backend.
  • JavaScript: controla la logica del frontend en el navegador.
  • HTML: define la estructura de las interfaces.
  • CSS: da estilo visual a la aplicacion.
  • SQL: se usa para crear tablas, relaciones y consultas de MySQL.

5.2 Librerias del backend

  • Flask: framework web ligero que permite crear endpoints y servir archivos.
  • flask-cors: autoriza la comunicacion entre frontend y backend cuando estan en dominios diferentes.
  • flask-mysqldb: conecta la aplicacion con MySQL.
  • flask-bcrypt: protege contrasenas usando hash.
  • flask-jwt-extended: implementa autenticacion basada en tokens JWT.
  • gunicorn: servidor WSGI usado para despliegue en produccion.

5.3 Librerias del frontend

  • Bootstrap 5: acelera la construccion de pantallas adaptables.
  • Bootstrap Icons: provee iconos.
  • SweetAlert2: mejora la presentacion de mensajes y confirmaciones.

5.4 Herramientas y entorno

  • MySQL: gestor de base de datos relacional.
  • VS Code: editor sugerido por la configuracion local del proyecto.
  • venv: entorno virtual para aislar dependencias Python.
  • Render: plataforma usada para desplegar la API.
  • Netlify: plataforma usada para desplegar el frontend.

6. Estructura general de carpetas

  • backend/: logica del servidor.
  • frontend/: vistas, scripts y estilos de la interfaz.

7. Analisis de la carpeta backend

7.1 Archivo principal

app.py es el corazon del backend. En este archivo se crea la aplicacion Flask, se cargan los parametros de configuracion y se registran los modulos de rutas.

Funciones academicamente importantes que se observan aqui:

  • Inicializacion de un framework web.
  • Carga de configuracion por variables de entorno.
  • Integracion con JWT y MySQL.
  • Definicion de rutas visuales como /login, /pos y /admin.
  • Habilitacion de CORS para soportar frontend y backend separados.

7.2 Configuracion

config.py concentra parametros sensibles como claves secretas y datos de MySQL. Esto enseña una buena practica: separar configuracion del codigo fuente.

7.3 Utilidades del backend

  • requirements.txt: muestra las dependencias de Python.
  • generate_password.py: genera hashes para contrasenas.
  • fix_admin_hash.py: permite actualizar el hash del administrador.
  • api_stdout.log y api_stderr.log: sirven para depuracion.

7.4 Carpeta backend/routes

Las rutas no contienen toda la logica; su funcion es enrutar la solicitud hacia el controlador correcto.

  • auth_routes.py: endpoint de login.
  • product_routes.py: endpoints de productos, categorias, inventario y operaciones avanzadas.
  • sale_routes.py: endpoint para registrar ventas.
  • cash_register_routes.py: endpoints de caja.
  • user_routes.py: endpoints CRUD de usuarios.

7.5 Carpeta backend/controllers

Esta capa es ideal para estudiar validacion, reglas de acceso y coordinacion del negocio.

  • auth_controller.py: verifica usuario, estado y contrasena; genera JWT.
  • product_controller.py: entrega productos y categorias, y permite crear, actualizar o eliminar productos.
  • sale_controller.py: valida el carrito y el metodo de pago antes de enviar la venta al modelo.
  • cash_register_controller.py: controla apertura y cierre de caja.
  • inventory_controller.py: administra filtros, resumenes y ajustes de inventario.
  • advanced_inventory_controller.py: maneja procesos mas completos como compras, conteos, auditoria y transferencias.
  • user_controller.py: administra usuarios y valida que el actor sea administrador.

7.6 Carpeta backend/models

Los modelos representan la capa de acceso a datos. Aqui es donde se entiende mejor como una aplicacion conversa con MySQL.

  • user_model.py: consultas sobre usuarios.
  • product_model.py: consultas y operaciones sobre productos y categorias.
  • sale_model.py: registra ventas usando transacciones; valida caja abierta y stock.
  • cash_register_model.py: maneja la tabla de cajas y calcula diferencias de cierre.
  • inventory_model.py: resume el inventario, aplica ajustes y registra movimientos.
  • advanced_inventory_model.py: contiene procesos empresariales avanzados con multiples tablas relacionadas.

8. Analisis de la carpeta frontend

El frontend es la parte que interactua directamente con el usuario. Aunque no usa frameworks modernos como React o Vue, resulta muy valioso para estudiar JavaScript puro, manipulación del DOM y consumo de APIs con fetch.

8.1 Archivos HTML

  • index.html: redirige a login.
  • login.html: formulario de autenticacion.
  • pos.html: interfaz de caja y ventas.
  • admin.html: panel administrativo.
  • inventory.html: panel de control de inventario.
  • operativa.html: panel de operaciones avanzadas.

8.2 Scripts JavaScript

  • config.js: resuelve la URL de la API segun el entorno.
  • auth.js: guarda el token, recupera la sesion, valida roles y protege rutas del frontend.
  • login.js: controla el envio del formulario de autenticacion.
  • pos.js: administra productos, carrito, metodos de pago, caja y checkout.
  • admin.js: administra dashboard, productos y usuarios.
  • inventory.js: consulta KPIs, movimientos y widgets de alto impacto.
  • operativa.js: gestiona compras, recepcion, conteos, transferencias y auditoria.
  • ui.js: encapsula toasts, alertas, tema visual y modo simple o avanzado.

8.3 CSS compartido

app-shell.css contiene estilos base comunes. Desde el punto de vista del diseño de software, esto evita repetir reglas visuales y ayuda a mantener coherencia entre pantallas.

9. Analisis de la carpeta database

La carpeta database muestra como evoluciona un sistema de informacion a traves de migraciones. Esto es importante porque en la vida real una base de datos no se queda estatica: crece con las necesidades del negocio.

  • schema.sql: base de ventas, detalle, caja y movimientos de inventario.
  • migration_users.sql: usuarios y administrador inicial.
  • migration_user_status.sql: agrega estado activo/inactivo.
  • migration_add_cash_register.sql: integra caja con ventas.
  • migration_inventory_control.sql: agrega stock minimo y trazabilidad.
  • migration_high_impact.sql: agrega modulo avanzado de inventario.

10. Funcionamiento detallado del sistema

10.1 Autenticacion

  1. El usuario digita correo y contrasena en login.html.
  2. login.js envia la solicitud POST /api/login.
  3. auth_controller.py recibe los datos y consulta el usuario.
  4. La contrasena se compara con bcrypt.
  5. Si es correcta, se crea un token JWT.
  6. El frontend guarda la sesion en localStorage.

10.2 Consulta y filtrado de productos

  1. La interfaz llama a /api/products y /api/categories.
  2. El controlador verifica autenticacion.
  3. El modelo consulta MySQL y devuelve datos serializados.
  4. El frontend construye tarjetas, tablas y filtros.

10.3 Registro de ventas

  1. El usuario arma un carrito en pos.js.
  2. Selecciona metodo de pago.
  3. El sistema exige caja abierta.
  4. La API valida los datos.
  5. El modelo realiza una transaccion SQL.
  6. Se insertan encabezado y detalle de la venta.
  7. Se descuenta stock de cada producto.

Este flujo es excelente para estudiar integridad transaccional.

10.4 Apertura y cierre de caja

  1. El usuario define un monto inicial.
  2. La API crea un registro en la tabla cajas.
  3. Al cerrar, el sistema suma las ventas asociadas.
  4. Calcula monto esperado, monto contado y diferencia.

10.5 Inventario

El inventario no solo muestra stock. Tambien calcula resumenes, detecta agotados, productos bajos y valor monetario del inventario. Ademas, registra cada ajuste en una tabla historica, lo cual aporta trazabilidad.

10.6 Operaciones avanzadas

La pantalla operativa concentra procesos mas cercanos a un ERP o sistema empresarial:

  • Generacion de ordenes de compra.
  • Recepcion parcial o total de mercancia.
  • Registro de fechas de vencimiento por lote.
  • Conteos fisicos y conciliacion.
  • Transferencias entre sucursales o bodegas.
  • Auditoria de acciones realizadas.

11. Que aprende un estudiante de ADSI con cada modulo

  • Login: autenticacion, hash de contrasenas, sesiones y seguridad.
  • POS: carrito, formularios, eventos y operaciones transaccionales.
  • Admin: CRUD, roles y gestion de datos.
  • Inventario: trazabilidad, filtros, reportes y reglas de negocio.
  • Operativa: modelado de procesos empresariales y tablas relacionadas.
  • Base de datos: llaves primarias, foraneas, indices y consistencia.

12. Glosario de terminos

  • API: interfaz que permite que dos sistemas se comuniquen.
  • Endpoint: URL especifica de la API.
  • JWT: token usado para autenticar usuarios.
  • CRUD: crear, leer, actualizar y eliminar datos.
  • Hash: transformacion segura de una contrasena.
  • Transaccion: conjunto de operaciones SQL que deben completarse juntas.
  • Rollback: deshacer cambios de una transaccion cuando ocurre un error.
  • Modelo relacional: organizacion de datos en tablas conectadas.
  • Trazabilidad: capacidad de saber que cambio se hizo, cuando y por quien.
  • Migracion: cambio estructural aplicado a la base de datos.

13. Preguntas de autoestudio

  1. Que diferencia existe entre una ruta, un controlador y un modelo.
  2. Por que es importante usar JWT en vez de confiar solo en la interfaz.
  3. Que problema evita una transaccion en el registro de ventas.
  4. Por que conviene guardar movimientos historicos de inventario.
  5. Que ventajas ofrece separar configuracion en config.py.
  6. Como se relacionan las tablas ventas y detalle_venta.
  7. Que diferencia hay entre stock actual, stock minimo y stock contado.
  8. Por que un usuario inactivo no deberia poder iniciar sesion.

14. Ruta recomendada para estudiar este proyecto

  1. Empieza por backend/app.py para ubicar el punto de entrada.
  2. Estudia el login completo desde frontend hasta modelo.
  3. Analiza el flujo de venta desde pos.js hasta sale_model.py.
  4. Revisa el CRUD de productos y usuarios.
  5. Estudia el modulo de caja.
  6. Continua con inventario basico.
  7. Finaliza con el modulo avanzado de operativa.

15. Conclusion general

PuntoFacil POS es un excelente ejemplo de estudio porque refleja problemas reales del desarrollo de software: autenticacion, control por roles, transacciones, persistencia, mantenimiento, auditoria y crecimiento por modulos. Como estudiante de ADSI, revisar este proyecto con cuidado te permite pasar de la teoria a una estructura real de aplicacion.

En otras palabras, este sistema te ayuda a responder preguntas fundamentales de programacion de software: donde entra la informacion, donde se valida, donde se transforma, donde se guarda y como se presenta al usuario final.
